### Q: "How do you handle real-time updates with sub-100ms latency?"
**Member 2 Answer:**
> "Great technical question! We use several strategies:
>
> **1. WebSocket Connections:** Persistent bidirectional connections between client and server eliminate HTTP handshake overhead. Once connected, updates are instant.
>
> **2. Event-Based Architecture:** Instead of broadcasting every update to every user, we use game-based channels. Valorant players only receive Valorant match updates. This reduces server load by 80%.
>
> **3. Connection Pooling:** We efficiently manage thousands of concurrent WebSocket connections using Node.js's event loop and connection pooling.
>
> **4. Optimistic UI Updates:** Client shows changes immediately before server confirms, giving perception of zero latency.
>
> **5. Strategic Caching:** Frequently accessed data (gamer profiles, game lists) cached client-side to avoid round trips.
>
> In practice, match request updates appear in <50ms, which is imperceptible to human reaction time."

## Category 5: Development & Learning (All Members)

### Q: "What was the biggest technical challenge?"
**Member 3 Answer (primary), others can add:**
> "The biggest challenge was cross-origin authentication with real-time WebSockets:
>
> **The Problem:**
> - Frontend on Vercel (vercel.app domain)
> - Backend on Railway (railway.app domain)
> - Browsers block cookies across domains for security
> - WebSockets need authentication to know which user is connected
>
> **The Solution Required:**
> 1. Configure CORS with credentials allowed
> 2. Use SameSite=None cookies with Secure flag
> 3. Handle OAuth redirects across domains
> 4. Maintain WebSocket authentication state
> 5. Deal with preflight OPTIONS requests
>
> **What We Learned:**
> - Deep understanding of browser security models
> - HTTP cookie mechanics (SameSite, Secure, HttpOnly)
> - OAuth 2.0 flows in distributed environments
> - WebSocket authentication patterns
> - Debugging browser DevTools network tab religiously
>
> It took 3-4 days of intense debugging, reading RFCs, and trial-and-error. But we got it working rock-solid."

**Member 2 can add:**
> "That challenge taught us distributed system authentication is non-trivial. In a monolith, auth is easy. In modern multi-cloud architectures, you need to deeply understand HTTP, cookies, and browser security."

## General Q&A Tips & Techniques

### Handling "I Don't Know":
> "That's an excellent question that goes beyond our current implementation. Based on what we know, we'd approach it by [educated guess]. But we'd want to research that properly before committing to a specific answer. Would you like us to follow up with details after the presentation?"

### Handling Off-Topic Questions:
> "That's an interesting direction, though it's outside the scope of our current MVP. For this project, we focused on [core objectives]. But that would be an excellent area for future research or expansion."

### Handling Technical Questions Beyond Your Depth:
> "Great technical question. Let me think through this systematically... [pause to think] Based on [relevant principle], we'd approach it by [reasoned answer]. If this were a production critical issue, we'd also consult [documentation/expert/research] to ensure optimal implementation."

### Handling Cost/Price Objections:
> "I understand the concern about [aspect]. Let's look at this pragmatically: [show numbers/data]. When you compare [our approach] to [alternative], the value becomes clear. Plus, at our scale, [explain economies of scale/efficiency]."

### Handling Hostile or Skeptical Questions:
1. **Pause** (don't react emotionally)
2. **Acknowledge** ("I hear your concern about...")
3. **Clarify if needed** ("Just to ensure I understand correctly...")
4. **Respond with data** (facts, numbers, evidence)
5. **Stay confident** (you built something impressive)
6. **Redirect positively** ("What we've proven is...")
